## プロダクトビジョン

**OpsHub** は、中〜大規模組織向けの業務統合SaaSである。
申請承認ワークフロー、案件管理、工数管理、経費精算、請求管理、ドキュメント管理を
単一プラットフォームで提供し、業務の可視化・効率化・監査対応を実現する。

### 解決する課題
1. **業務の分断**: 申請・工数・経費・請求がバラバラのツールで管理され、情報の統合に手間がかかる
2. **承認の遅延**: 紙やメールベースのワークフローによるリードタイムの長さ
3. **監査対応の困難**: 「誰が・いつ・何を」の追跡に時間がかかる
4. **可視化の不足**: プロジェクト横断の工数・コスト把握がリアルタイムにできない

### 前バージョン (Next.js + Supabase) からの移行理由

| 課題 | 前バージョンの問題 | 新アーキテクチャでの解決 |
|---|---|---|
| **ビルド不安定** | turbopack issue が規模拡大で頻発 | Nx のキャッシュ + 安定したビルドパイプライン |
| **テスト不足** | Server Actions のユニットテストが困難 | NestJS TestingModule でサービス単体テスト容易 |
| **型安全の限界** | Supabase Client の型生成に依存 | Prisma V6 の型生成 + 共有型ライブラリ |
| **モジュール境界** | App Router のコロケーションのみ | Nx enforce-module-boundaries で依存制御 |
| **バグの後追い** | RLS エラーが空結果で返り検知困難 | NestJS Guards + Prisma Middleware で明示的エラー |

## スコープ

### IN（対象範囲）

| Epic | 領域 | 概要 | 優先度 |
|---|---|---|---|
| A | テナント/組織/権限 | マルチテナント、ユーザー管理、RBAC | **Must** |
| B | ワークフロー | 汎用的な申請→承認/差戻しフロー | **Must** |
| C | 案件/タスク/工数 | プロジェクト管理、タスク、工数入力/集計 | **Must** |
| D | 経費 | 経費申請、承認、集計、締め、証憑管理 | **Should** |
| E | 請求 | 請求書作成、ステータス管理、PDF/帳票 | **Should** |
| F | ドキュメント | ファイルアップロード/管理/共有 | **Could** |
| G | 通知/検索/レポート | 通知配信、全文検索、レポート/ダッシュボード | **Could** |

### OUT（対象外）
- 決済・会計仕訳連携（外部システム連携は将来検討）
- AI機能（外部AI APIの従量課金なし前提）
- モバイルネイティブアプリ（レスポンシブWebで対応）
- SSO/SAML連携（将来課題）

## 成功指標

| 指標 | 目標値 | 測定方法 |
|---|---|---|
| 申請リードタイム | 30%短縮 | ワークフロー作成〜承認完了の平均時間 |
| 月次締め作業 | 50%削減 | 経理の月次集計所要時間 |
| 監査追跡 | 5分以内 | 特定操作の who/when/what を特定する時間 |
| 工数入力率 | 95%以上 | 月間工数入力済みユーザー比率 |

## 技術前提（概要）

| 項目 | 選定 | 根拠 |
|---|---|---|
| モノレポ管理 | Nx 20.x | プロジェクトグラフ + affected コマンド + キャッシュ |
| フロントエンド | Angular 19 (Standalone) | Signal ベース + Vite 開発サーバー |
| UI | Angular Material 19 | エンタープライズ向け豊富なコンポーネント |
| バックエンド | NestJS 10 | DI + モジュール構成 + SWC 高速ビルド |
| ORM | Prisma V6 | 型安全クエリ + マイグレーション管理 |
| 開発DB | SQLite | セットアップゼロ、高速テスト |
| 本番DB | PostgreSQL | 本番運用の安定性 |
| 認証 | JWT (Passport.js) | NestJS 統合 |
| 認可 | NestJS Guards + Prisma Middleware | RBAC + テナント分離 |
| テスト | Vitest + Playwright | ユニット/統合/E2E |
